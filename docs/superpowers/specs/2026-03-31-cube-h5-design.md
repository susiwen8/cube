# Cube H5 版本设计

## 目标

在现有魔方项目基础上实现一个手机优先的 H5 正式版本，支持独立部署，并尽量复用当前共享核心能力。H5 首版需要覆盖现有小程序的核心功能集合，包括：

- `3-10` 阶切换
- 手势转层与视角拖拽
- 打乱、复位、自动求解与回放
- 教学模式
- 计时与本地成绩
- 高阶内层记号展示与高阶求解提示

H5 页面不复刻当前小程序布局，而是基于移动网页使用习惯重新设计，并尽量兼容较广范围的移动浏览器和老一些的 WebView 环境。

## 范围与约束

### 已确认的产品约束

- 功能范围：完整版本，而非精简 Demo
- 主要终端：手机优先
- 视觉与交互：重新设计，不直接复刻小程序
- 兼容目标：尽量广兼容，包含较老 Android WebView / 低端机型
- 技术栈：允许引入前端框架
- 发布形态：独立站点，可部署到静态托管或 CDN

### 非目标

- 首版不做 WebGL / 第三方 3D 引擎迁移
- 首版不做云端排行榜
- 首版不做账号体系
- 首版不做 PWA / 离线缓存
- 首版不做分享链路或社交裂变能力

## 实现方案对比

### 方案 A：React H5 壳，直接复用 shared core/render

优点：

- 实施速度最快
- `src/shared/core`、`src/shared/render`、`app-controller` 可直接复用
- H5 页面可快速独立建设

缺点：

- 现有 `miniapp-page` 中的运行时逻辑需要在 Web 端重写一份
- 三端后续容易产生行为分叉

### 方案 B：抽出共享 runtime，再接 React H5 壳

优点：

- 三端共享更多逻辑，长期维护成本最低
- H5 与双小程序在播放、动画、视口、触摸解释等行为上更容易保持一致
- 后续新增平台时复用基础最好

缺点：

- 首次改造范围更大
- 需要先整理现有小程序页面运行时

### 方案 C：只复用 core，Web 侧重写 render/interaction

优点：

- H5 自由度最高
- 未来做更激进的网页体验最灵活

缺点：

- 重复建设最多
- 风险最大
- 不适合作为当前完整 H5 首版

## 推荐方案

采用 `方案 B`：抽出共享 runtime，再接 `React + Vite` H5 壳。

原因：

- 当前项目已经具备较强的共享核心与共享渲染能力
- H5 是正式版本而非轻量展示页，需要控制长期维护成本
- 共享 runtime 可以避免把最复杂的交互与播放循环维护成两套

## 目标架构

项目从“共享内核 + 双小程序壳”升级为“共享内核 + 共享 runtime + 三端壳”。

### 共享核心

- `src/shared/core`
  - 魔方状态、转动、打乱、求解、教学、计时
- `src/shared/render`
  - 投影、场景构建、命中检测、手势映射、Canvas 绘制

这两层继续作为三端共同依赖的基础能力，不引入平台特有 API。

### 新增共享 runtime

新增 `src/shared/runtime`，从 `src/shared/adapters/miniapp-page.js` 中提取平台无关的页面运行时逻辑，主要负责：

- 维护视口状态（`yaw` / `pitch`）
- 管理当前手势状态、命中结果与拖拽锁定
- 管理动画状态与播放队列
- 驱动单循环调度
- 协调 `app-controller` 与渲染层
- 组装供 UI 层消费的视图状态

### 平台壳

- `miniapps/wechat`
- `miniapps/alipay`
- `apps/h5`

三端各自只保留平台能力差异：

- 生命周期接入
- Canvas 上下文初始化
- 事件标准化
- 存储适配
- UI 结构与样式

## H5 技术方案

### 应用结构

H5 采用 `React + Vite` 构建独立站点，目录建议为：

- `apps/h5`
  - `src/main.*`
  - `src/App.*`
  - `src/components/*`
  - `src/hooks/*`
  - `src/styles/*`
  - `index.html`
  - `vite.config.*`

React 负责：

- 页面结构
- UI 状态展示
- 事件绑定
- 移动端布局与样式

React 不负责：

- 魔方状态机
- 自动求解业务
- 渲染几何计算
- 手势到转动的最终业务解释

这些继续通过共享 runtime 与共享核心承担。

### 页面结构

H5 首版采用单页应用布局，分为三层：

1. 顶部状态带
   - 当前阶数
   - 当前模式
   - 计时
   - 步数
   - 状态文案
2. 中部主舞台
   - 魔方 Canvas
   - 手势操作区域
   - 当前提示文案
3. 底部可折叠控制面板
   - 常用操作按钮
   - 模式说明
   - 教学内容
   - 成绩列表
   - 求解队列与策略

### 交互设计

- 贴纸拖拽用于转层
- 空白区拖拽用于旋转视角
- 常用操作按钮保持大尺寸，适合手机触控
- 教学说明、成绩、求解信息进入底部抽屉，避免首屏信息拥挤
- 高阶模式下显示内层记号提示，如 `2R / 3U / 4F`

## 数据流设计

H5 采用单向数据流：

1. 用户触摸 Canvas 区域
2. Web 事件桥接层将浏览器事件转换为共享 runtime 可识别的输入格式
3. runtime 调用：
   - `hit-test`
   - `gesture-mapper`
   - `app-controller`
4. `app-controller` 更新 session
5. runtime 根据 session 与动画状态重建 scene
6. React UI 根据 view model 刷新页面显示
7. Canvas 组件根据 scene 执行绘制

这样可以保持 React 只是界面层，而共享 runtime 与共享核心仍然是唯一业务真相源。

## 兼容策略

### 浏览器能力

为兼容较广范围的移动浏览器，H5 采用以下策略：

- 渲染坚持使用 `2D Canvas`
- 同时支持 `Pointer Events` 与 `Touch Events`
- 浏览器支持 `Pointer` 时优先使用 `Pointer`
- 不支持时退回 `touchstart / touchmove / touchend`
- 避免依赖重型第三方手势库

### 页面触控控制

- 设置适合移动端的 `viewport`
- 在魔方交互区尽量减少浏览器原生滚动与双击缩放干扰
- 使用可控的事件监听选项处理拖拽阶段默认行为

### 存储兼容

- 默认使用 `localStorage`
- 读写失败时回退到内存存储
- 当无法持久化时，只提示当前会话可用，不阻塞核心玩法

### Canvas 兼容

- 采用 `CSS 尺寸 + devicePixelRatio` 缩放
- 对高 DPR 设备设置上限，避免低端机绘制压力过大
- 初始化失败时展示可理解的错误态，而非白屏

## 性能策略

- 保留单循环调度思想，避免多个定时器竞争
- H5 侧优先使用 `requestAnimationFrame`
- 当页面进入后台或不可见时暂停连续回放
- 高阶模式减少不必要的重绘和装饰开销
- 若拖拽过程中锁定 move 未变化，不重复重绘相同场景

## 错误处理

### 初始化异常

- Canvas 创建失败：展示错误提示与刷新建议
- 存储不可用：降级为会话内成绩
- 尺寸测量失败：使用默认正方形舞台尺寸兜底

### 运行时异常

- 非法事件数据：回退为空输入，不让页面崩溃
- 回放队列为空：自动停止播放状态
- 页面切后台：暂停自动播放，避免时间与状态漂移

### 用户可感知反馈

- 重要异常给出轻量可读提示
- 不把底层报错直接暴露给用户
- 核心操作失败时优先允许用户继续复位或重试

## 测试策略

测试继续坚持“共享逻辑优先、平台适配分层验证”的原则。

### 继续保留的现有测试

- `core` 行为测试
- `render` 几何与手势测试
- `app-controller` 与集成流程测试

### 新增测试层

1. 共享 runtime 单测
   - 视口拖拽
   - move 锁定
   - 动画完成
   - 回放推进
   - 计时刷新
   - 页面状态刷新
2. H5 适配测试
   - Pointer / Touch 标准化
   - `localStorage` 读写与降级
   - Canvas DPR 缩放
   - 页面可见性切换
3. H5 集成测试
   - 打乱 -> 操作 -> 自动求解
   - 计时 -> 完成 -> 成绩记录
   - 阶数切换 -> 高阶回退求解

## 实施阶段

### 第一阶段：抽出共享 runtime

- 从 `miniapp-page` 中提取平台无关运行时
- 保证现有小程序测试继续通过
- 让小程序端先接入新 runtime

### 第二阶段：搭建 H5 应用

- 新建 `apps/h5`
- 接入共享 runtime
- 完成 Canvas、触摸交互、模式切换、成绩持久化

### 第三阶段：移动端体验打磨

- 完成新 H5 视觉布局
- 处理广兼容细节
- 补足 H5 端测试与文档

## 成功标准

H5 版本完成后，应满足以下条件：

- 可独立部署为站点
- 手机浏览器中可完成完整核心玩法
- `3-10` 阶全部可切换
- 教学、计时、成绩、自动求解全部可用
- 现有微信 / 支付宝小程序能力不回退
- 共享逻辑复用率高，不出现明显三端分叉
